5. Deeper Understanding Statement

The list immediately below includes all items produced that are beyond what we produced in labs.

  1. Procedural geometry. No pre-existing .glb files were used to create any of the items; all were created at run time by the browser. I explicitly referenced a list of 2D points (13 points for cans; 18 for bottle) in the source and fed the array of numbers into THREE.LatheGeometry; this takes the information and revolves it around the Y-axis to create a 3D object. By using this method, I was able to fine-tune the silhouette one point at a time, and the same function produces both the cans and the bottle.
  2. Procedural textures. No labels were image files either; all were generated in JavaScript, written directly to an HTML <canvas> and converted to a texture to be used with Three.js. Three editions are available for the label (standard, night and retro 80s), and changing labels simply redraws the canvas in a different colour scheme. No PNGs are provided via network access.
  3. Custom GLSL shader. The glass used with Nova Cola includes a Three.js ShaderMaterial, with a vertex and fragment shader created by me for this specific project (assets/js/shaders.js). The fragment shader has a Fresnel term calculated as a function of the view direction and surface normal. This results in an opaque bottle along the edge at some angles, while the middle of the bottle is still transparent enough to see the liquid mesh inside.
  4. Post-processing. The rendered model is sent to a Three.js EffectComposer, which includes a RenderPass, UnrealBloomPass and OutputPass. This chain of processing enables a bloom pass used to provide a visual glow (soft lighting) around the brightest high points of the amber glass bottle and the metallic surfaces. This bloom pass was wired to the "Bloom" button, so visitors can see what the scene looks like with and without it.
  5. PHP + SQLite backend. An additional feature of the About page is a small guestbook form located at the bottom of the page. When completed and submitted, this form creates a JSON formatted request body and sends it to api/feedback.php, where a PHP process writes the submitted data row into a SQLite database using PDO. Should the page be displayed using the VS Code Live Server and the PHP server is off, the same JavaScript code will function and use localStorage instead of sending the request, so none of the guestbook submissions will appear to have failed during testing.
  6. MVC organisation. The JavaScript of the application follows a pattern generally known as model-view-controller (MVC). In this case, the model is a handful of state variables at the top of app.js together with a data/products.json file that stores all the products and product information, which is loaded to populate those variables. The view is the visual presentation of the product and packaging; the Three.js Viewer class located within the scene.js file generates all necessary visual elements, and all DOM updates from within app.js are triggered after an AJAX request has been completed. The controller of the application is a collection of event handlers placed within the app.js file to capture click, key press and form submit events. All products are loaded into the application via AJAX requests to the JSON data file, so when adding or modifying product information, HTML does not need to be touched.
  7. Accessibility. Access to the three primary product actions can be accomplished with keyboard shortcuts. The keyboard shortcut for the three product options (one for each product) is designated by 1/2/3. The shortcut for wireframe product displays (turning wireframe on/off) is designated by W. The shortcut for auto-rotation functionality is designated by R, and the single spin is designated by S. A visible focus indicator (default browser behaviour) indicates which button currently has focus, so that users can clearly identify what control they are focused on. The CSS associated with the application reflects the prefers-reduced-motion media query, and the hero button transition is set to "none". Each product thumbnail located within the product gallery has an aria-label property, which allows a screen reader user to identify which can is which.

Testing feedback 

I performed testing of the site on Chrome 147, Firefox 147 and Edge 147 on Windows 11, and on Safari 19.3 on macOS Sequoia. In all browsers, all three products loaded successfully, the wireframe and bloom buttons functioned as described, the three label revisions were interchanged without the need to refresh, and the feedback data submitted via the feedback form was inserted into the feedback.sqlite database after the PHP server was started.

I encountered a few issues that required updates during the most recent test cycle.

The first issue observed during testing was that the CanvasTexture for the labels was appearing much lighter in colour on the Safari browser compared to Chrome's results. After additional testing, it was determined that the Safari browser interprets a Canvas as a linear colour space. Therefore, I assigned the tex.colorSpace value to THREE.SRGBColorSpace for every instance of a texture. This resulted in all three browsers rendering the colours with appropriate consistency.

The second issue observed during testing was on Mobile Safari. When the Mobile Safari address bar was hidden, then reappeared, the Three.js canvas was not resizing to match the viewport. I added a window resize listener, which calls the renderer.setSize() and composer.setSize() methods, which seems to have resolved the issue.
